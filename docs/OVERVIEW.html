Project overview · v0.2.0 · walkthrough & tech map

AVANZARE

A recruiter's resume-screening desktop app. Point it at a folder of CVs or a hiring inbox, define the keywords that matter, let a local or cloud LLM rank the survivors against your job description — and send every decision with a human in the loop.

Electron desktop engine ⁄ UI split local-first data Ollama or Claude SQLite talent pool
A

How it's built

Two packages in one npm-workspaces monorepo. All the real logic lives in a headless engine that knows nothing about Electron; the desktop app is a thin shell that drives it over IPC. That split is what makes a future background/server mode cheap.

packages/engine

Core engine

Parsing, OCR, keyword filtering, contact extraction, LLM clients, SMTP, IMAP, SQLite, Excel export, error framework. Plain async functions over a database — no UI.

IPC ⇄ envelope
packages/desktop

Electron shell

Main process owns the engine and exposes it over typed IPC; a context-isolated preload bridge; a hand-built React UI with dark/light theming.

B

The workflow, stage by stage

A single screening run moves through six stages. Stages 3–6 are the pipeline; the two shaded steps are where you stay in control — rescuing near-misses and confirming every email. The Batch tab runs stages 1–3 for many jobs at once, then hands each finished job back to stage 4.

1

Technical Setup

views/Setup.tsx · config/profiles.ts

Before the app opens you pick a settings profile: CV source (local folder or IMAP inbox), the LLM provider & endpoint, OCR, SMTP, email templates, export folder and concurrency. Save named profiles (one per client) and mark one "use automatically" to skip this next time. Passwords are encrypted on the machine.

JSON profile store Electron safeStorage Test-connections probes
2

Define the screening job

views/Job.tsx

Set the job title, mandatory keywords (each weighted 1–5), nice-to-have keywords, optional synonyms, requirement tags (certifications / experience range / publications), a hiring target, and the detailed job description the LLM will judge against. For an email source you also pick the date range to import.

React controlled forms weighted-keyword model
3

Parse & keyword-screen

pipeline/screening.ts

Every CV is read to text, the applicant's name/email/phone are extracted and saved to the talent database, keywords are matched with word-boundary awareness, and each application is bucketed: rejected (missing a mandatory keyword), mandatory-only, or mandatory + optional. All of it is persisted, so nothing is re-parsed later.

pdfjs-dist 3 mammoth / word-extractor tesseract.js (OCR fallback) in-house mapLimit pool SHA-256 CV hash
4

Rejection review

views/Rejection.tsx

The applicants that failed the keyword filter appear in a table sorted by keyword score, best near-miss first. Every row starts checked = will be rejected. Uncheck anyone to rescue them into the LLM stage. Edit contact info inline, open a CV preview, see a candidate's cross-job history, and add internal notes.

Human checkpoint. Rejection emails are only sent after you approve a preview and a 30-second undo elapses — never automatically.
5

LLM analysis

llm/router.ts · llm/ollama.ts · llm/anthropic.ts

Accepted and rescued CVs are scored 0–100 against your job description, with formal-education assessment and a ✓/✗ verdict per requirement tag — returned as structured JSON, not prose, so results are reliable even with small local models.

Ollama HTTP API @anthropic-ai/sdk JSON-schema structured output
6

Results & emails

views/Results.tsx · mail/mailer.ts

A ranked table by affinity score with tier, education, requirement verdicts and expandable reasoning. Every row starts unchecked — check the ones you want to advance. One button emails everyone: checked get the acceptance template, unchecked the rejection template.

Human checkpoint. Because everyone in the table is emailed, the send goes through the preview modal + hard confirmation + 30-second undo.
Nodemailer / SMTP {{placeholder}} templates

CV sources

Recursively scans a folder (subfolders included) for .pdf / .docx / .doc. Network drives and synced cloud folders work by pointing the path at them.

Parsing & OCR

PDF text extraction

parsing/extract.ts

Reads PDFs page by page, reconstructing line breaks from hasEOL so the contact heuristics have real lines to work with. Uses the CJS legacy build for stable Node behaviour.

pdfjs-dist 3 (legacy)

Word extraction

parsing/extract.ts

Modern .docx via mammoth's raw-text extraction; legacy .doc via word-extractor.

mammothword-extractor

OCR for scanned PDFs

parsing/ocr.ts

When a PDF has no text layer and OCR is on, each page is rasterized at 2× and read by Tesseract. The heavy deps load lazily, so only scanned CVs pay for them.

tesseract.js 7@napi-rs/canvaspdfjs rasterizelazy dynamic import

Contact extraction

parsing/contact.ts

Heuristics pull the first email-like token, phone-like digit runs, and a plausible name near the top — with the IMAP sender's address/name overriding the guess when present.

custom regex heuristics

Screening & matching

Weighted keyword matching

pipeline/keywords.ts

Case-insensitive matching with word boundaries applied only next to alphanumeric edges, so java ≠ javascript but c++ and .NET match. Each mandatory keyword's 1–5 importance drives the keyword score.

custom boundary regexweighted scoring

Keyword synonyms

pipeline/keywords.ts (buildSynonymMap)

A keyword matches if it or any alias appears, but the canonical term is what's recorded — so "Amazon Web Services" counts for "AWS" without skewing tiers, scores or the dashboard.

synonym map

Concurrency pool

util/concurrency.ts

A tiny bounded mapLimit (replacing p-queue) caps how many CVs process at once; the LLM stage is additionally capped at 2 in-flight requests.

in-house mapLimit

Batch / multitasking

pipeline/batch.ts · views/Batch.tsx

Queue several screening tasks — each with its own profile and keywords — and run them concurrently. Per-task progress arrives on dedicated IPC channels; each finished task hands off to the normal review wizard.

Good to know

Where your data lives

  • avanzare.sqlite — candidates, jobs, applications, email log
  • profiles/*.json — settings (passwords encrypted)
  • email-cvs/<run>/ — CVs downloaded from a mailbox
  • logs/avanzare.log — JSON-lines log
  • All under %APPDATA%/avanzare-desktop/

Privacy & compliance

  • Emails are never sent automatically — you confirm every batch
  • Ollama keeps CVs on your network; Claude sends CV text to Anthropic
  • Per-candidate purge honors erasure requests
  • The audit trail is built for GDPR / EEOC discovery

Known edges

  • Tesseract fetches language data from a CDN on first OCR use
  • Cloud sources (Drive/OneDrive/S3) are stubbed behind AVZ-SRC-403
  • The batch queue is in-memory (runs persist; the queue list doesn't)
  • The installer is unsigned — SmartScreen shows a warning

Error codes at a glance

  • PARSE 1xx — text extraction & OCR
  • LLM 2xx — Ollama / Claude analysis
  • MAIL 3xx · SRC 4xx — email & sources
  • CFG 5xx · DB 6xx · EXP 7xx — config, database, export
